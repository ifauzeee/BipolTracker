Latar Belakang

Permasalahan Utama

  • Ketidakpastian Jadwal: Mahasiswa menunggu tanpa estimasi waktu yang jelas.
  • Inefisiensi Waktu: Waktu terbuang hanya untuk menunggu bus di halte.
  • Resiko Keselamatan: Potensi kebocoran gas/asap pada armada tua tanpa deteksi dini.

Urgensi Penelitian

Diperlukan sistem cerdas yang mengintegrasikan IoT dan Mobile Web untuk memberikan transparansi posisi armada dan pemantauan kondisi keselamatan secara real-time.

Dasar Teori

Global Positioning System

Menggunakan prinsip Trilaterasi dengan minimal 4 satelit untuk menentukan koordinat presisi (Latitude, Longitude, Altitude).

IoT Architecture

Menerapkan model komunikasi Machine-to-Machine (M2M) dengan protokol ringan (UDP) untuk efisiensi bandwidth pada jaringan seluler GPRS.

Tujuan & Manfaat

Transparansi

Menyediakan informasi lokasi dan estimasi kedatangan bus secara akurat.

Keselamatan

Mendeteksi anomali kadar gas berbahaya pada kabin bus.

Manajemen

Monitoring kinerja operasional armada bagi pengelola kampus.

Metodologi Pengembangan

Ringkasan Eksekutif

IoT Full-Stack

Sistem pelacakan real-time end-to-end yang mengintegrasikan perangkat keras ESP32, backend Node.js, dan frontend PWA modern.

Latensi Rendah

Menggunakan protokol UDP untuk transmisi data perangkat dan WebSockets untuk distribusi klien, meminimalkan delay hingga <1 detik.

Integrasi Menyeluruh

Sistem BIPOL dirancang dengan pendekatan Modular Layered Architecture untuk memastikan modularitas dan skalabilitas.

  • Modularitas: Pemisahan jelas antara perangkat keras (Edge) dan perangkat lunak (Cloud).
  • Skalabilitas: Backend berbasis Node.js mampu menangani ribuan koneksi konkuren via WebSockets.
  • Reliabilitas: Penggunaan protokol UDP untuk transmisi data sensor menjamin overhead data yang minim.

Spesifikasi Hardware

Microcontroller Unit (MCU)

  • ESP32 DevKit V1: Dual-core 32-bit LX6, 240 MHz.
  • Memory: 520 KB SRAM, 4 MB Flash.
  • Connectivity: WiFi 802.11 b/g/n & Bluetooth 4.2.

Modul Komunikasi & Sensor

  • SIM808 (GSM+GPS): GPRS Class 12, GPS Accuracy < 2.5m CEP.
  • MQ-2 Gas Sensor: Deteksi LPG, Propana, Hidrogen (Range 300-10000 ppm).
  • I2C LCD 16x2: Interface visual driver lokal.

Bedah Teknis: Firmware

Inti Perangkat Keras

ESP32 sebagai kontroler utama + SIM808 untuk konektivitas GPS/GPRS.

Alur Logika

Looping BipolTracker_V2.ino membaca stream NMEA setiap detik.

Protokol Optimal

Data dikirim via UDP sebagai string CSV (id,lat,lon,speed,gas) untuk efisiensi byte.

Sensor Keselamatan

Integrasi Sensor MQ-2 untuk deteksi dini kebocoran gas/asap di bus.

Bedah Teknis: Backend

Jembatan Layanan UDP

Modul udpService.js bertindak sebagai pendengar socket mentah. Modul ini mem-parsing CSV, memvalidasi geolokasi (mencegah koordinat 0,0), dan menyuntikkan data ke pipeline database.

Penyimpanan Supabase

Penyimpanan PostgreSQL dengan tabel bipol_tracker yang bersifat append-only untuk jejak audit rute historis.

Keamanan & Integritas

Secure Headers

Implementasi modul Helmet.js untuk mencegah serangan umum seperti XSS dan Clickjacking.

Anti-Spam

Sistem Rate Limiting membatasi bombardir request pada endpoint pelaporan publik.

Sanitasi Data

Modul HPP (HTTP Parameter Pollution) mencegah manipulasi query database via URL.

Payload Control

Limitasi ukuran body JSON (Maks 10KB) melindungi server dari beban overload data sampah.

MapLibre GL

Modul Vanilla JS

Pengujian QoS

Quality of Service (QoS) mengukur performa dan keandalan sistem komunikasi real-time pada aplikasi BIPOL Tracker.

Apa itu QoS?

Quality of Service adalah pengukuran kualitas layanan jaringan untuk memastikan sistem dapat mengirim dan menerima data secara cepat, stabil, dan andal.

Kecepatan Keandalan Skalabilitas

Metode Pengujian

  • UDP Socket Testing (Port 3333)
  • WebSocket/Socket.IO (Port 3000)
  • Node.js Automated Testing Scripts
--real (ESP32) --dummy (Simulasi)

Pengujian yang Dilakukan

1. Latency Test

Mengukur waktu respons end-to-end dari pengiriman data GPS hingga diterima di browser

2. Throughput Test

Mengukur kapasitas pengiriman paket UDP per detik dan total data yang bisa dikirim

3. Concurrent Connection Test

Mengukur kemampuan server menangani banyak koneksi WebSocket secara bersamaan

4. Reliability Test

Mengukur keandalan pengiriman data jangka panjang dan packet loss rate

Metode Pengujian

Automated Testing

Pengujian dilakukan menggunakan Node.js scripts custom yang dirancang untuk berinteraksi langsung dengan backend server via UDP dan WebSocket.

Automation & Consistency

Menjalankan skenario pengujian yang sama berulang kali (repeatable) dengan parameter yang konsisten.

High Precision & Real-time

Menghitung metrik presisi tinggi (ms) dan menangkap data live tanpa overhead rendering browser.

Flexible Modes

Mendukung mode --real untuk monitoring data asli dan --dummy untuk stress testing load tinggi.

qos_latency_test.js 
const socket = io('http://localhost:3000');

// Listen for real-time bus updates
socket.on('update_bus', (data) => {
    const receiveTime = Date.now();
    
    if (USE_REAL_DATA) {
        // Mode REAL: Calculate interval between packets
        if (lastRealDataTime) {
            const interval = receiveTime - lastRealDataTime;
            stats.latencies.push(interval);
        }
        lastRealDataTime = receiveTime;
        
        console.log(` ${data.lat}, ${data.lon} |  ${data.speed} km/h`);
    }
});

Cara Menjalankan:

node tests/qos_latency_test.js --real

Tantangan & Solusi

Kendala Sinyal

Masalah: Blank spot di area gedung parkir menyebabkan data loss.

Solusi: Firmware menyimpan buffer lokal dan mengirim ulang paket saat sinyal kembali (Store & Forward).

Beban Server

Masalah: Koneksi WebSocket memakan memori tinggi.

Solusi: Implementasi interval update adaptif (berubah lambat saat bus berhenti).
